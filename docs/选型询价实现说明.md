# 选型询价实现说明

这份文档描述的是当前仓库里已经落地的“北泽产品选型 + 询价”实现，不是未来规划稿。

## 1. 功能范围

当前项目已经具备下面这些能力：

- 北泽二级分类初始化
- 北泽一级分组信息回填到二级类目
- 按分类查看选型产品
- 基于规格字段做逐步筛选
- 从选型结果发起询价
- 用户查看自己的询价单并留言
- 管理员查看询价单、录入报价、更新状态
- 管理员维护系统公共设置与报价单模板
- 通过 `modelNo -> Model.partNumber` 自动匹配 3D 模型

## 2. 主要代码入口

### 前端

- `client/src/data/beizeSelection.ts`
  北泽分类蓝图，包含：
  - 一级分组和二级类目
  - `selectionFields`
  - `kind`
  - `detailTemplate`
  - 少量 `exampleModels`

- `client/src/pages/SelectionPage.tsx`
  选型主页面。流程是：
  - 先选一级分组
  - 再选二级类目
  - 根据 `selectionFields` 逐步筛选
  - 展示结果卡片
  - 支持加入询价、跳模型详情、提交建模/技术支持

- `client/src/pages/SelectionAdminPage.tsx`
  选型后台管理。支持：
  - 分类增删改
  - 产品增删改
  - JSON 批量导入产品
  - 维护一级分组和类目归组

- `client/src/pages/MyInquiriesPage.tsx`
  用户查看自己的询价单列表。

- `client/src/pages/InquiryDetailPage.tsx`
  询价单详情，支持：
  - 用户取消询价
  - 双方留言
  - 管理员报价
  - 管理员接受/拒绝状态流转

- `client/src/pages/SettingsPage.tsx`
  系统设置页。支持站点标题、Logo、公开配置、备份恢复等公共设置维护。

- `client/src/pages/QuoteTemplateEditor.tsx`
  报价单模板编辑页。支持拖拽调整区块顺序、开关标准区块、编辑自定义区块，并把模板保存到系统设置中的 `quote_template`。

### 后端

- `server/src/routes/selections.ts`
  选型分类和产品接口。

- `server/src/routes/inquiries.ts`
  询价单、询价留言、管理员报价和状态接口。

- `server/src/routes/settings.ts`
  系统设置、图片上传、备份恢复、版本检测等接口，同时对外提供公开设置和报价模板配置。

- `server/prisma/schema.prisma`
  选型与询价相关的数据表定义。

- `server/prisma/seed-beize.ts`
  从 `docs/北泽选型结构.json` 初始化全部北泽分类。

- `data/seeds/products/batch*.ts`
  产品数据批次脚本（batch1 ~ batch79），每个负责一个类目的一段连续页码。存放在项目 `data/` 目录，不随应用源码发布。

- `server/prisma/seeds/run-batch.ts`
  批次通用入口，指向 `data/seeds/products/`，用法：`npm run prisma:seed:batch 30`。

- `data/seeds/backfill-groups.ts`
  给已存在的北泽二级类目回填 `groupId / groupName / groupIcon / kind`。

- `data/seeds/backfill-sort-types.ts`
  给已存在类目的 `columns` 补 `sortType`，用于前端更稳定地按数值或螺纹规则排序。

### 数据源

- `docs/北泽选型结构.json`
  当前分类初始化的唯一结构化来源。

- `docs/北泽选型清单.md`
  当前的业务整理依据，适合人工补录和后续扩充时参考。

## 3. 数据模型

### 3.1 选型分类 `SelectionCategory`

用途：

- 存二级类目
- 定义当前类目可展示的参数列
- 承载前端一级分组展示所需的分组信息
- 标记当前类目是普通商品、方案类还是知识表类

关键字段：

- `name`
- `slug`
- `description`
- `icon`
- `sortOrder`
- `columns`
- `image`
- `groupId`
- `groupName`
- `groupIcon`
- `kind`

说明：

- 当前数据库里仍然没有单独建一级分组表。
- 但一级分组展示信息已经直接回填到 `SelectionCategory` 的 `groupId / groupName / groupIcon`。
- `kind` 现在也已入库，当前用于区分 `product / solution / knowledge` 这几类页面形态。

### 3.2 选型产品 `SelectionProduct`

用途：

- 存一个二级类目下的可选产品或方案项

关键字段：

- `categoryId`
- `name`
- `modelNo`
- `specs`
- `image`
- `pdfUrl`
- `sortOrder`
- `isKit`
- `components`

说明：

- `specs` 是 JSON，对当前项目最关键。
- 前端筛选和结果展示主要依赖 `specs`。
- 套件类产品通过 `isKit + components` 表达。

### 3.3 询价相关

表结构分成三层：

- `Inquiry`
  询价单头信息，包含状态、联系人、总价、管理员备注

- `InquiryItem`
  询价明细，包含产品名、型号、规格、数量、单价

- `InquiryMessage`
  询价单留言，支持用户和管理员双向沟通

### 3.4 系统设置 `Setting`

用途：

- 存站点级公共设置
- 存公开可读配置
- 存报价单模板 JSON

当前和选型 / 询价流程直接相关的设置包括：

- `selection_page_title`
- `selection_page_desc`
- `field_aliases`
- `quote_template`

## 4. 状态和业务流

当前询价状态流：

- `submitted`
- `quoted`
- `accepted`
- `rejected`
- `cancelled`

其中：

- 用户创建询价时，状态直接为 `submitted`
- 管理员提交报价后，状态变为 `quoted`
- 管理员可把 `quoted` 更新为 `accepted` 或 `rejected`
- 用户只能取消 `submitted` 或 `draft`

注意：

- Prisma 里没有单独的状态枚举，当前是字符串状态，后续如果状态继续扩展，前后端都要同步。

## 6. 分类初始化和产品导入

### 6.1 初始化分类

当前北泽分类初始化命令：

```bash
cd server
npm run prisma:seed:beize
```

它会读取：

- `docs/北泽选型结构.json`

并 upsert 全部北泽二级类目。

### 6.2 导入产品数据

当前已经存在一批按画册页码拆分的脚本：

- `data/seeds/products/batch1.ts`
- `data/seeds/products/batch2.ts`
- ...
- `data/seeds/products/batch79.ts`

另有两个一次性整理脚本：

- `server/prisma/seeds/backfill-groups.ts`
- `server/prisma/seeds/backfill-sort-types.ts`

运行方式：

```bash
cd server

# 运行单个批次（例如 batch30）
npm run prisma:seed:batch 30

# 运行分类初始化
npm run prisma:seed:beize

# 运行回填脚本
npm run prisma:seed:backfill:groups
npm run prisma:seed:backfill:sort
```

这些批次脚本的共同约束：

- 先通过 `CATEGORY_SLUG` 找到目标类目
- 按 `categoryId + modelNo` 查重
- 已存在则更新，不存在则创建
- `specs` 中保留核心筛选字段
- 建议把 `图册来源` 保留在 `specs` 中，便于回溯页码
- `sortOrder` 要稳定，避免同类产品反复导入后顺序抖动

### 6.3 批量 JSON 导入的定位

后台页里的批量导入更适合：

- 临时补少量产品
- 导入已经整理好的结构化 JSON

不太适合：

- 画册大批量、长期维护的数据录入

原因：

- 批次脚本更容易复跑
- 批次脚本能做规则推导
- 批次脚本便于保留页码来源和统一排序

## 7. 当前实现里的几个重要约束

### 7.1 前端蓝图和数据库不是完全同一份信息

当前有两层来源：

- 数据库 `SelectionCategory.columns`
- 前端 `client/src/data/beizeSelection.ts`

页面实际逻辑是：

- 优先用数据库里的 `columns`
- 没有时回退到前端蓝图里的 `selectionFields`

这意味着：

- 如果只改前端蓝图，不重新 seed 分类，线上类目列定义可能不会变
- 如果只改数据库列定义，前端蓝图里的 `detailTemplate/exampleModels` 仍然需要单独维护

### 7.2 当前是“部分入库，部分仍在前端蓝图”

当前这些信息已经入库：

- `groupId`
- `groupName`
- `groupIcon`
- `kind`

当前这些信息仍只存在前端蓝图里：

- `detailTemplate`
- 一部分 `exampleModels`

所以现在数据库层已经能判断某个类目是：

- 普通商品
- 方案类
- 知识表类

但数据库层还不能单独决定详情页模板和示例型号展示，后续如果后台也要完整接管这些内容，建议继续把对应配置补进 `SelectionCategory` 或独立配置表。

### 7.3 模型匹配依赖 `modelNo`

产品列表接口会按：

- `SelectionProduct.modelNo`
- `Model.partNumber`

做自动匹配。

所以新增产品数据时，`modelNo` 最好保持和模型库料号一致，否则前端不会自动出现“模型”入口。

## 8. 备份与恢复系统

### 8.1 功能范围

后台管理页面提供完整的备份与恢复功能：

- 一键创建全量备份（数据库 + 文件）
- 从已有备份列表恢复
- 从上传的 `.tar.gz` 备份文件恢复
- 备份文件下载、重命名、删除
- 恢复前自动创建安全快照，失败自动回滚

### 8.2 核心代码

- `server/src/lib/backup.ts` — 备份/恢复核心逻辑
- `server/src/routes/settings.ts` — 备份恢复 HTTP 接口
- `server/src/lib/settings.ts` — `clearSettingsCache()` 供恢复后清缓存

### 8.3 备份覆盖范围

备份内容：

| 内容 | 归档路径 | 说明 |
|------|----------|------|
| 数据库全量 dump | `_backup_db/database.sql` | `pg_dump --no-owner --no-privileges` |
| 备份元数据 | `_backup_db/meta.json` | 时间戳、版本号、应用版本、已备份目录列表 |
| 上传元数据 | `_backup_db/metadata/` | `uploads/.metadata/` 的完整副本 |
| 模型文件 | `models/` | 已转换的 3D 模型 |
| 缩略图 | `thumbnails/` | PNG 预览图 |
| STEP 原始文件 | `originals/` | 用户上传的原始 CAD 文件 |
| 产品图纸 | `drawings/` | PDF 图纸文件 |
| 选项图片 | `option-images/` | 选型产品的展示图片 |
| 工单附件 | `ticket-attachments/` | 工单/反馈附件 |
| Logo | `logo/` | 站点 Logo 文件 |
| Favicon | `favicon/` | 站点图标文件 |
| 水印 | `watermark/` | 水印图片文件 |

备份格式为 `.tar.gz` 归档，打包时自动排除 `__MACOSX`、`.DS_Store`、`._*` 等无关文件。

### 8.4 元数据格式 `meta.json`

```json
{
  "timestamp": "2025-04-25T10:30:00.000Z",
  "version": "2.0",
  "appVersion": "2.1.0",
  "staticDirs": ["models", "thumbnails", "originals", "drawings", "logo"]
}
```

- `version: "2.0"` — v2.0 格式，包含扩展目录和上传元数据
- `appVersion` — 备份时的应用版本号（从 `package.json` 读取）
- `staticDirs` — 本次备份实际包含的静态目录列表（只备份当时存在的目录）

### 8.5 安全快照机制

恢复操作在执行任何破坏性操作（`DROP SCHEMA CASCADE`）之前，会**强制**导出当前数据库的安全快照：

```
恢复流程：
1. 解压备份归档，提取 database.sql
2. SQL 预处理（过滤不兼容语句）
3. ★ 创建安全快照（pg_dump 当前数据库）— 失败则中止恢复
4. 重置 schema → 导入备份数据
5. 如果导入失败 → 自动回滚到安全快照
6. 应用数据库迁移（prisma migrate deploy，失败时用 prisma db push 兜底）
7. 恢复文件（staging 模式：先提取到临时目录，成功后再替换原目录）
8. 恢复 uploads/.metadata
9. 清理所有缓存
```

关键保障：

- **安全快照是强制的**：如果无法创建快照，整个恢复操作中止，不做任何破坏
- **自动回滚**：数据库导入失败时自动用安全快照恢复原状
- **回滚也有降级**：如果安全快照回滚也失败，会尝试重建空 schema + 迁移

### 8.6 文件恢复的 staging 模式

文件恢复采用 staging 目录模式，避免中途失败导致数据丢失：

1. 提取归档中的目录到 `.restore_xxx/` 临时目录
2. 提取成功后删除原目录
3. 将 staging 目录重命名为目标目录
4. 如果提取失败，原目录保持不变

对于 `uploads/.metadata` 的恢复，额外使用 backup/rollback 模式：
- 先将现有 `.metadata` 重命名为 `.metadata_backup`
- 然后把提取的元数据移入
- 如果移入失败，从 backup 恢复原数据

### 8.7 并发保护

通过文件锁防止同一时间多个备份/恢复操作并发执行：

- 锁文件：`uploads/.backup_restore.lock`
- 包含 PID 和时间戳
- 如果锁文件超过 2 小时，视为过期锁自动清理

### 8.8 缓存清理

恢复完成后自动清理所有缓存：

- Redis 缓存：清除所有 `cache:` 前缀的缓存键
- 设置缓存：调用 `clearSettingsCache()` 清除内存中的设置缓存

### 8.9 兼容性处理

- 旧版备份（无 `_backup_db/` 前缀）：自动尝试 `database.sql` 和 `meta.json`
- 旧版备份（缺少扩展目录如 `option-images/`）：跳过，保留当前数据不删除
- 从"服务器文件"恢复时，`removeAfter = false`，不删除原始上传文件
- `prisma migrate deploy` 失败时自动降级到 `prisma db push --skip-generate`

### 8.10 Docker 环境

备份系统同时支持本地 PostgreSQL 和 Docker 容器中的 PostgreSQL：

- 启动时检测本地 `pg_dump` 是否可用
- 不可用时自动检测运行中的 `postgres` 容器，通过 `docker exec` 执行命令
- 数据库文件通过 `docker cp` 传入容器执行恢复

## 9. 后续继续做时的推荐顺序

推荐优先顺序：

1. 补完当前已开始录入但未闭环的类目页码
2. 再录入不锈钢接头和阀门
3. 最后处理方案类和知识表类

原因：

- 当前前五个 slug 已经有脚本模式，可以直接复用
- 接头类参数结构最稳定
- 方案类后续可能需要和普通商品不同的详情页展现

## 10. 新增批次脚本时建议保持的格式

建议一个脚本只负责：

- 一个 `CATEGORY_SLUG`
- 一段连续或相近页码

建议保留这些常量：

- `CATEGORY_SLUG`
- `SOURCE_REF` 或每组数据里的 `sourceRef`
- 公共材质 / 介质 / 表面处理常量

建议每条产品至少写入这些字段：

- `name`
- `modelNo`
- `sortOrder`
- `specs.型号`
- 当前类目的筛选字段
- `specs.图册来源`

## 11. 和当前代码对应时要注意的地方

- 目前 `SelectionPage` 支持“逐步筛选”和“直接搜索”两种模式
- 目前 `SelectionPage` 也支持先按一级分组进入，再选二级类目
- 当前搜索只查 `name` 和 `modelNo`
- 公开设置里的 `selection_page_title / selection_page_desc / field_aliases` 会直接影响选型页展示
- 询价创建时会优先从 `productId` 回填产品名、型号和规格

- 报价打印页会读取 `quote_template`，所以模板编辑器保存后的结果会直接影响报价单打印布局
