# 并发压测与部署调优

这份文档用于验证“浏览模型、加载静态资源、下载模型文件”三类主要压力。线上承载能力不要只看在线人数，要看同一时刻的活跃请求数、下载带宽、数据库连接、Redis 状态和 Node/API CPU。

## 快速压测

先启动完整容器环境：

```bash
docker compose -f docker-compose.local.yml up -d --build
```

浏览接口压测：

```bash
BASE_URL=http://localhost:3780 \
SCENARIO=browse \
CONCURRENCY=200 \
DURATION_SECONDS=60 \
node scripts/load-test.mjs
```

静态资源压测：

```bash
BASE_URL=http://localhost:3780 \
SCENARIO=static \
CONCURRENCY=500 \
DURATION_SECONDS=60 \
node scripts/load-test.mjs
```

下载链路压测：

```bash
BASE_URL=http://localhost:3780 \
SCENARIO=download \
MODEL_ID=你的模型ID \
FORMAT=original \
NO_RECORD=1 \
CONCURRENCY=100 \
DURATION_SECONDS=60 \
node scripts/load-test.mjs
```

`NO_RECORD=1` 适合压测，避免压测数据污染下载历史和下载次数。要验证真实下载记录落库时，去掉这个变量；下载记录会进入 Redis 队列并由后台 worker 批量写库，通常 1-5 秒内出现在后台。

如果线上开启了登录下载，传入登录后的 access token，让脚本先创建一次性下载令牌再下载：

```bash
ACCESS_TOKEN=你的JWT \
MODEL_ID=你的模型ID \
SCENARIO=download \
CONCURRENCY=100 \
node scripts/load-test.mjs
```

混合压测会同时覆盖浏览、静态和少量下载：

```bash
SCENARIO=mixed MODEL_ID=你的模型ID CONCURRENCY=300 DURATION_SECONDS=120 node scripts/load-test.mjs
```

## 观察指标

压测过程中同时观察：

```bash
docker stats
docker compose logs -f api web
docker exec 3dparthub-postgres pg_isready -U modeluser -d 3dparthub
docker exec 3dparthub-redis redis-cli info stats
```

建议目标：

- 浏览接口：错误率低于 1%，p95 在 300-800ms 内，数据库连接没有打满。
- 静态资源：主要压力应落在 Nginx，API 容器 CPU 不应明显升高。
- 下载文件：Nginx 通过 `X-Accel-Redirect` 接管传输，API 只负责鉴权和记录，不应长时间占用 Node 流式传输。
- 混合场景：如果 p95 快速升高，先看数据库连接池、Redis、Nginx 带宽和磁盘 IO。

## 生产参数

推荐从保守值起步：

```env
API_WORKERS=2
DB_CONNECTION_LIMIT=5
CONVERSION_WORKER_CONCURRENCY=1
```

更高并发时再逐步提高：

- `API_WORKERS` 通常设置为 CPU 核数或核数的一半，转换任务较重时不要把 API worker 开太满。
- `DB_CONNECTION_LIMIT * API_WORKERS` 要小于 PostgreSQL 可用连接数，预留维护连接。
- `CONVERSION_WORKER_CONCURRENCY` 影响 CPU 和内存，浏览下载为主时建议低一点，避免转换任务抢资源。
- 大文件下载主要受公网带宽和磁盘 IO 限制，不是 Node 并发数本身。

## 判断是否能承受几千人

几千“在线用户”通常不会等于几千“同时请求”。如果只是浏览和下载：

- 静态模型预览、缩略图、前端资源由 Nginx 直接服务，承载能力主要看带宽。
- 下载接口由 API 鉴权后交给 Nginx 发送文件，API 压力较轻。
- 真正需要压测的是同时活跃请求，例如 200、500、1000、2000 并发阶梯。

压测顺序建议：

1. `browse` 从 100 并发开始，逐步到 500、1000。
2. `static` 从 500 并发开始，观察 Nginx 和带宽。
3. `download` 根据模型文件大小从 20、50、100 并发开始，不要一上来把本机磁盘或公网带宽打满。
4. `mixed` 用接近真实比例的并发跑 10 分钟以上。
