# 代码维护规范

本项目按“小批量、可验证、不中断行为”的方式长期维护。重构优先保持接口、权限和数据结构不变，先搬移边界，再清理内部实现。

## 后端分层

- `server/src/routes`: 只处理 HTTP 入参、状态码和响应，不承载复杂业务流程。
- `server/src/services`: 放业务编排、数据库读写、文件处理和跨模块流程。
- `server/src/lib`: 放通用能力，例如配置、校验、缓存、队列、权限辅助、下载令牌。
- `server/src/middleware`: 只放 Express 中间件，例如认证、限流、审计、错误处理。
- `server/src/workers`: 放后台任务入口，业务细节应继续下沉到 `services`。

新增接口时优先复用：

- `asyncHandler`: 包装异步路由，把异常交给全局错误处理中间件。
- `HttpError`: 表达可预期 HTTP 错误。
- `requestValidation`: 解析字符串、分页、ID、开关等请求参数。

## 前端分层

- `client/src/pages`: 页面级组合，避免堆积复杂表单、表格和弹窗逻辑。
- `client/src/components`: 可复用 UI 组件；页面私有组件可先放同名子目录。
- `client/src/hooks`: 放可复用状态逻辑、查询逻辑和副作用。
- `client/src/api`: 只封装 API 请求和响应类型，不放页面状态。
- `client/src/lib`: 放纯函数工具，例如格式化、排序、模板处理、安全清洗。
- `client/src/types`: 放跨页面共享类型。

## 文件大小目标

- 普通工具和组件：建议 300 行以内。
- 页面文件：建议 500 行以内。
- 后端路由文件：建议 500 行以内。
- 超过 800 行的文件进入拆分候选清单。

拆分大文件时先按“接口域/功能域”拆，不先做抽象。重复代码出现 2-3 次后再提公共函数。

## 路由规范

推荐结构：

```ts
router.get("/api/example", authMiddleware, asyncHandler(async (req, res) => {
  const { page, pageSize, skip } = paginationQuery(req.query);
  const items = await exampleService.list({ page, pageSize, skip });
  res.json({ items, page, page_size: pageSize });
}));
```

路由里避免：

- 直接复制分页解析逻辑。
- 大段数据库事务和文件系统流程。
- 手写重复的 400/401/403/404 错误分支。
- 在一个文件里混放公开接口、管理员接口、上传接口和后台任务接口。

## 验证要求

每批重构至少执行：

```bash
cd server && npm run typecheck
cd server && npm test
cd client && npm run typecheck
```

涉及 Docker、Nginx 或构建流程时追加：

```bash
docker compose config --quiet
cd client && npm run build
```

## 分批顺序

1. 公共规范和后端基础工具。
2. 拆 `server/src/routes/models.ts`。
3. 拆 `server/src/lib/backup.ts`。
4. 拆 `client/src/pages/SettingsPage.tsx`。
5. 拆 `client/src/pages/ModelAdminPage.tsx`。
6. 拆选型页面和选型管理页面。
7. 拆 3D 预览组件。
8. 全局一致性收尾。
